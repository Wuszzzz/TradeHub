# TradeHub 架构文档

## 1. 架构方向

TradeHub 当前采用“大服务、强模块边界、关键能力进程级解耦”的技术路线：

- 不在第一阶段做完整微服务治理。
- 允许服务按能力解耦为独立进程，例如基金后端、股票后端、行情网关、前端。
- 保留一个统一前端、一个基金后端、一个股票后端、一个行情网关。
- 在 `tradehub-stock` 内部补齐股票数据平台、量化引擎、交易模拟、AI 研究等模块边界。
- 在数据库、任务、OpenAPI、统一响应、错误码、日志审计层面先建立地基。

这条路线的目标是：能承接 `pythonstock/stock`、`myhhub/stock`、`ArvinLovegood/go-stock` 的功能规模，但不让早期架构被完整微服务体系的复杂度拖慢。

## 3. 模块边界

| 模块 | 边界 | 不应该承担 |
| --- | --- | --- |
| `tradehub-frontend` | 路由、交互、图表、表格、状态展示 | 第三方行情直连、复杂指标批量计算 |
| `tradehub-fund` | 基金、账户、持仓、认证、通知、基金 AI 报告 | 股票行情采集、股票指标计算 |
| `tradehub-stock` | 股票自选、行情聚合、指标、形态、选股、回测、告警、模拟交易、股票 AI 研究 | 第三方源原始协议细节 |
| `tradehub-market-api` | 第三方行情源适配、缓存、singleflight、防击穿、源健康 | 用户业务、策略、回测、持仓 |
| PostgreSQL | 业务配置、账户、任务、策略、审计、结果索引 | 高频行情明细 |
| TDengine | K 线、分钟线、逐笔、资金流、指标序列 | 用户配置、权限、策略定义 |
| Redis | 缓存、锁、限流、任务短状态 | 长期业务数据 |

## 4. 第一阶段地基

第一阶段优先事项：

1. 修正 README 和本地启动文档，统一真实运行方式。
2. 建立 OpenAPI 契约，按 `fund`、`stock`、`market`、`quant`、`ai`、`system` 分域。
3. 统一 API 响应结构、错误码、分页、批量任务返回格式。
4. 梳理数据库分层，明确 PostgreSQL、TDengine、Redis、文件存储职责。
5. 把 `tradehub-stock/api` 从单包大文件逐步拆成接口层、应用层、领域层、基础设施层。
6. 建立统一任务模型，支持采集、指标、K 线形态、选股、回测、AI 报告。

详细方案见 [docs/技术架构/1. 第一期架构地基迭代方案.md](./docs/技术架构/1.%20第一期架构地基迭代方案.md)。

## 5. 解耦边界

第一阶段允许以下解耦：

| 解耦对象 | 保留原因 |
| --- | --- |
| 前端独立进程 | Vite 开发、桌面/移动封装、统一 UI 迭代 |
| 基金后端独立进程 | Django/DRF 已有成熟账户、持仓、通知、AI 配置体系 |
| 股票后端独立进程 | Go 更适合行情、任务、指标、回测等计算型模块 |
| 行情网关独立进程 | 第三方源适配、缓存、singleflight、防击穿应和业务解耦 |

第一阶段不做以下治理：

| 不做 | 原因 |
| --- | --- |
| 服务注册中心 | 当前服务数量少，固定本地端口和环境变量足够 |
| 服务网格 | 运维成本高，不能直接提升股票/基金功能交付 |
| 分布式事务 | 业务边界可通过任务和幂等处理 |
| 多仓库拆分 | 会降低需求文档、API、前端联动效率 |
| 过早 Kubernetes 化 | 当前关键问题是功能和数据地基，不是弹性调度 |

迁移原则：

- 先拆包，不改业务行为。
- 先抽接口和 DTO，再迁移实现。
- 旧接口路径保持兼容。
- 每迁移一个模块就补一个最小测试。

## 7. 数据分层

| 层 | 存储 | 典型数据 |
| --- | --- | --- |
| source/raw | 文件或 PostgreSQL raw 表 | 第三方原始响应、源请求审计 |
| staging | PostgreSQL staging 表 | 清洗前结构化数据、字段映射结果 |
| curated/time-series | TDengine | K 线、分钟线、逐笔、资金流、指标序列 |
| business | PostgreSQL | 自选、账户、策略、任务、告警、回测结果索引 |
| cache | Redis | 实时快照、短 TTL 排行、任务锁、限流 |
| artifact | 本地文件或对象存储 | AI 研报、回测 HTML/CSV、导出文件 |

详细方案见 [docs/技术架构/3. 数据库分层设计.md](./docs/技术架构/3.%20数据库分层设计.md)。

## 8. API 契约方向

统一响应：

```json
{
  "success": true,
  "code": "OK",
  "message": "ok",
  "data": {},
  "meta": {
    "request_id": "req_xxx",
    "timestamp": "2026-06-14T12:00:00+08:00"
  }
}
```

分域路径：

| 域 | 路径前缀 |
| --- | --- |
| 基金 | `/api/fund/v1/*` |
| 股票 | `/api/stock/v1/*` |
| 行情 | `/api/market/v1/*` |
| 量化 | `/api/quant/v1/*` |
| AI 研究 | `/api/ai/v1/*` |
| 系统 | `/api/system/v1/*` |

现有接口可保留兼容，新增能力按新路径和统一响应推进。
