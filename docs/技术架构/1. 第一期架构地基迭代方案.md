# 第一期架构地基迭代方案

## 1. 目标

第一期不先写页面，不先堆股票菜单功能。先把 TradeHub 的技术地基补齐，保证后续复刻 `pythonstock/stock`、`myhhub/stock`、`ArvinLovegood/go-stock` 时不会把行情、指标、选股、回测、AI、交易全部写成页面直连和接口堆叠。

第一期产出：

| 产出 | 目标 |
| --- | --- |
| README 和本地启动文档 | 统一当前真实运行方式 |
| OpenAPI 契约 | 统一接口分域、版本、路径、请求/响应 |
| 统一 API 响应 | 统一成功、失败、分页、批量任务、错误码 |
| 数据库分层 | 明确 PostgreSQL、TDengine、Redis、文件存储职责 |
| stock API 拆包方案 | 把股票后端从工具型 API 迭代为可扩展大模块 |
| 任务模型 | 支撑采集、指标、形态、选股、回测、AI 报告 |

## 2. 核心原则

- 可以做服务解耦，但不做完整微服务化治理；先做大服务内部模块化和关键能力进程级解耦。
- 页面不是第一优先级，API 契约和数据结构先行。
- 行情源能力收敛到 `tradehub-market-api`，业务逻辑收敛到 `tradehub-stock`。
- 计算型能力走任务系统，不能让页面请求同步跑大批量计算。
- 股票和基金共享基础设施，但业务表、任务、领域模型保持边界清晰。
- 新接口使用统一响应，旧接口先兼容，逐步迁移。

## 3. 第一阶段工作拆分

### 3.1 文档和启动基线

| 工作 | 文件 | 验收标准 |
| --- | --- | --- |
| 修正根 README | `README.md` | 不再描述旧 `services/*` 路径和全 Docker 本地启动 |
| 新增本地启动文档 | `LOCAL_DEV.md` | 能按文档启动 infra、fund、stock、market、frontend |
| 重写架构基线 | `ARCHITECTURE.md` | 明确大服务模块化路线 |
| 标注 compose 边界 | `README.md` / `LOCAL_DEV.md` | 明确当前本地以 `dev.sh` 为准 |

### 3.1.1 服务解耦边界

| 服务 | 是否独立进程 | 第一阶段定位 |
| --- | --- | --- |
| `tradehub-frontend` | 是 | 统一前端壳和业务页面 |
| `tradehub-fund` | 是 | 基金业务、认证、通知、基金 AI |
| `tradehub-stock` | 是 | 股票业务、量化、回测、股票 AI |
| `tradehub-market-api` | 是 | 行情源适配和缓存 |
| `quant-engine` | 否 | 先作为 `tradehub-stock` 内部模块，不单独拆进程 |
| `ai-research` | 否 | 先作为 fund/stock 内部应用模块和统一配置，不单独拆服务 |

第一阶段不引入服务注册中心、服务网格、跨服务事务、独立 API Gateway。服务之间通过固定环境变量和本地端口协作。

### 3.2 OpenAPI 与统一响应

| 工作 | 输出 | 验收标准 |
| --- | --- | --- |
| 统一路径分域 | OpenAPI 文档 | 新增接口按 `/api/{domain}/v1/*` 命名 |
| 统一响应结构 | 响应规范 | 成功、失败、分页、批量任务格式一致 |
| 统一错误码 | 错误码表 | 前端能按错误码做稳定处理 |
| 统一 request_id | API 规范 | 日志、响应、任务审计可串联 |

### 3.3 数据库分层

| 工作 | 输出 | 验收标准 |
| --- | --- | --- |
| 业务数据分层 | 数据库分层文档 | 每类数据有明确存储位置 |
| 股票表域规划 | 表域清单 | 自选、行情索引、指标、形态、策略、回测、任务、AI 分开 |
| TDengine 口径 | 时序模型 | K 线、分钟线、逐笔、资金流、指标序列可落库 |
| Redis 口径 | 缓存/锁规范 | 短 TTL 快照、任务锁、限流不混用长期数据 |

### 3.4 stock API 模块化拆包

| 工作 | 输出 | 验收标准 |
| --- | --- | --- |
| 目标目录结构 | 拆包方案 | 有接口层、应用层、领域层、基础设施层 |
| 迁移顺序 | 迁移计划 | 可逐模块迁移，不破坏旧接口 |
| DTO/契约抽离 | contract 包 | 前端和服务端字段稳定 |
| 任务模块抽离 | task application | 采集、指标、回测、AI 都能复用任务状态机 |

## 4. 推荐实现顺序

### 第 1 步：冻结运行事实

先让 README、`LOCAL_DEV.md`、`ARCHITECTURE.md` 与真实仓库一致。

关键事实：

- 基础设施：Docker。
- 业务服务：宿主机。
- 前端端口：`5173`。
- 基金 API：`7001`。
- 股票 API：`7002`。
- 行情网关：`17080`。
- PostgreSQL：`5433`。
- Redis：`6380`。
- TDengine：`16042`。

### 第 2 步：统一 API 合同

新增接口全部遵守：

```text
/api/fund/v1/*
/api/stock/v1/*
/api/market/v1/*
/api/quant/v1/*
/api/ai/v1/*
/api/system/v1/*
```

旧接口不立即删除，建立兼容层：

| 旧接口 | 处理 |
| --- | --- |
| `/api/v1/market/*` | 保留，内部转到 stock/market 应用服务 |
| `/api/v1/alerts/*` | 保留，新增 `/api/stock/v1/alerts/*` |
| `/api/v1/paper/*` | 保留，新增 `/api/stock/v1/paper/*` |
| `/api/health/` | 保留，后续补 `/api/system/v1/health` |

### 第 3 步：统一响应中间件

Go 和 Django 都要落同一种响应结构。

成功：

```json
{
  "success": true,
  "code": "OK",
  "message": "ok",
  "data": {},
  "meta": {
    "request_id": "req_20260614_xxx",
    "timestamp": "2026-06-14T12:00:00+08:00"
  }
}
```

失败：

```json
{
  "success": false,
  "code": "STOCK_SYMBOL_NOT_FOUND",
  "message": "标的不存在",
  "data": null,
  "meta": {
    "request_id": "req_20260614_xxx",
    "timestamp": "2026-06-14T12:00:00+08:00"
  },
  "error": {
    "field": "symbol",
    "detail": "6000000 is not a valid symbol"
  }
}
```

### 第 4 步：梳理数据层

数据先分层，再决定表。

| 数据类型 | 存储 |
| --- | --- |
| 用户、账户、权限 | PostgreSQL `fundval` |
| 股票自选、策略、告警、任务 | PostgreSQL `stock_etf` |
| K 线、分钟线、逐笔、资金流、指标序列 | TDengine `stock_etf_ts` |
| 实时快照、排行榜短缓存 | Redis |
| 回测报告、AI 研报、导出文件 | 本地文件或对象存储 |

### 第 5 步：拆 stock API 包结构

先迁移包结构，不改接口行为。

推荐顺序：

1. `response`、`errors`、`pagination`、`middleware`。
2. `instrument`、`watchlist`。
3. `market`。
4. `alert`。
5. `paper`。
6. `task`。
7. `quant`。
8. `backtest`。
9. `ai-research`。

## 6. 非目标

第一期不做：

- 不新增股票页面。
- 不重做 UI。
- 不把基金/股票拆成完整微服务体系。
- 不让前端直接访问第三方行情源。
- 不做全量历史行情大规模回填。
- 不一次性重写所有 stock API 代码。
