# Agent 能力中心架构设计

## 1. 背景与目标

TradeHub 现有 AI 能力主要集中在用户 AI 配置、提示词模板、AI 分析和报告预览。下一阶段需要新增一个大的能力类目：Agent 能力中心，用于承载多个独立智能体，每个 Agent 有自己的提示词、文章/知识内容、工具能力、规则约束、API 暴露方式和运行审计。

本设计覆盖三个核心域：

- Agent Skill：智能体可复用的技能包，例如基金分析、股票异动解释、公告解读、板块轮动分析。
- Agent Rules：智能体的行为规则、工具权限、输出约束、风控限制和上下文边界。
- Agent API：智能体对前端、定时任务、外部系统和内部服务暴露的统一调用 API。

设计目标：

- 支持创建 N 个 Agent，每个 Agent 架构独立，配置、提示词、文章内容、技能、规则和运行记录互相隔离。
- 支持每个子 Agent 维护自己的公众号、Markdown、文章、文件和外部链接，形成独立知识库。
- 支持动态识别知识来源类型、结构化抽取核心提示词、方法论、规则和工具需求。
- 支持把 Agent 标准化打包，提交到 Git、社区仓库或其他分发渠道，形成可复用的 Agent 社区。
- 支持从 Agent 提示词和文章内容中抽取工具需求，形成可审计、可绑定、可测试的工具清单。
- 支持 Agent 以同步、异步任务、会话式问答、定时任务四种方式运行。
- 支持后续股票、基金、板块、策略、资讯、回测等业务统一接入 Agent 能力。
- 保持和现有 `AIConfig`、`AIPromptTemplate` 兼容，避免一次性推翻已有 AI 设置。

```text

  ├── Agent 社区
  ├── Agent 运行工作台
  └── Agent 调用日志

  ├── /api/ai/v1/agents
  ├── /api/ai/v1/agent-knowledge
  ├── /api/ai/v1/agent-skills
  ├── /api/ai/v1/agent-rules
  ├── /api/ai/v1/agent-tools
  ├── /api/ai/v1/agent-packages
  ├── /api/ai/v1/agent-runs
  └── /api/ai/v1/agent-sessions

Agent Runtime
  ├── Context Builder
  ├── Prompt Renderer
  ├── Tool Planner
  ├── Rule Engine
  ├── Skill Executor
  ├── Model Gateway
  ├── Result Validator
  └── Audit Writer

Knowledge Pipeline
  ├── Source Detector
  ├── Content Parser
  ├── Prompt Extractor
  ├── Methodology Extractor
  ├── Tool Hint Extractor
  ├── Deduplicate / Versioning
  └── Package Builder

Tool Layer
  ├── 内置工具：基金、股票、行情、排行、板块、持仓、自选
  ├── MCP 工具：外部搜索、文件、数据库、浏览器、研报

  ├── PostgreSQL：配置、版本、运行、日志、审计

  └── 对象/文本存储：文章内容、长上下文、附件
```

## 3. 核心概念

### 3.1 Agent

Agent 是一个可独立配置和运行的智能体实例。每个 Agent 至少包含：

- 身份：名称、头像、角色、业务域、描述。
- 模型：默认 AI 配置、备用模型、Token 限额、温度参数。
- 提示词：系统提示词、任务提示词、输出格式提示词。
- 内容：文章、知识片段、样例报告、方法论、引用资料。
- 技能：绑定的 Agent Skill 列表。
- 规则：绑定的 Agent Rules 列表。
- 工具：允许调用的工具清单。
- API：可被外部调用的接口定义。
- 审计：运行记录、工具调用、成本、输出版本。

Agent 之间默认隔离，不能读取其他 Agent 的私有提示词、私有文章内容和私有运行上下文，除非显式配置共享。

### 3.2 Agent Skill

Skill 是 Agent 可复用的能力单元，建议按业务动作拆分，不按页面拆分。例如：

- `fund_profile_analysis`：基金资料与基金经理分析。
- `fund_sector_rotation`：基金板块归因与轮动分析。
- `stock_kline_review`：股票 K 线和指标复盘。
- `announcement_summary`：公告摘要与风险点抽取。
- `ranking_explain`：基金/股票排行变化解释。

Skill 本身不直接等于 prompt。它由输入 schema、输出 schema、默认提示词、工具依赖、规则依赖和运行器组成。

### 3.3 Agent Rules

Rules 是 Agent 的边界和约束，包括：

- 行为规则：必须引用来源、不能编造数据、必须说明数据日期。
- 工具规则：哪些工具可用、是否需要用户确认、超时和重试策略。
- 数据规则：可读取哪些用户数据、是否允许读自选/持仓/交易日志。
- 输出规则：Markdown/JSON/表格格式、免责声明、风险等级。
- 风控规则：不得直接下单、高成本任务二次确认、高风险结论必须给反向证据。

Rules 应版本化，运行时记录使用的规则版本。

### 3.4 Agent API

Agent API 是把 Agent 暴露给前端、定时任务和外部系统的契约。一个 Agent 可以有多个 API：

- `chat`：会话式问答。
- `run`：一次性任务执行。
- `stream`：流式执行。
- `schedule`：定时执行。
- `webhook`：外部事件触发。
- `tool-extract`：从提示词/文章中抽取工具需求。

### 3.5 Agent Knowledge

Agent Knowledge 是每个子 Agent 的私有或公开知识资产，来源可以是公众号、Markdown 文件、文章链接、PDF、研究笔记、历史报告、问答记录或 Git 仓库。知识不是简单全文拼接，而是要经过识别、解析、抽取、版本化和引用。

每个知识源至少需要识别：

- 来源类型：`wechat_mp` / `markdown` / `url_article` / `pdf` / `git_repo` / `manual_note` / `qa`。
- 来源身份：公众号名称、作者、URL、文件路径、仓库地址、commit hash。
- 内容结构：标题、摘要、正文、章节、表格、代码块、引用链接。
- 可抽取能力：核心提示词、方法论、规则、工具需求、样例输出、适用场景。
- 使用边界：私有/公开、是否可社区分发、是否允许模型训练式总结、是否需要保留原文链接。

### 3.6 Agent Package

Agent Package 是 Agent 社区流通的标准提交单元。一个 Agent Package 可以提交到 Git、社区仓库或其他分发渠道，内容包括 Agent 元信息、提示词版本、技能、规则、工具需求、知识索引和样例。

Package 是声明式的，不直接携带用户私有数据。私有持仓、自选、账户、API Key、Cookie、Token 不允许进入 Package。

## 5. 工具抽取机制

用户明确提到“抽取 Agent 提示词的工具信息”，这里需要作为独立能力设计。

### 5.1 输入来源

工具抽取来源包括：

- AgentPromptVersion：系统提示词、任务提示词、输出提示词。
- AgentContent：文章、方法论、样例报告。
- AgentSkill：技能说明和默认提示词。

### 5.2 抽取结果

抽取结果统一写为 Tool Hint：

```json
{
  "tool_code": "fund_latest_nav",
  "tool_name": "基金最新净值查询",
  "reason": "提示词要求引用最新净值和净值日期",
  "required": true,
  "input_fields": ["fund_code"],
  "output_fields": ["nav", "nav_date", "source"],
  "confidence": 0.92,
  "source_ref": {
    "type": "prompt",
    "id": "prompt_version_id",
    "fragment": "必须展示基金最新净值和更新时间"
  }
}
```

### 5.3 抽取流程

1. 用户保存 Agent 提示词或文章内容。
2. 系统触发 `extract_agent_tool_hints`。
3. 先用规则解析关键词，例如“净值、持仓、板块、公告、K线、排行、资金流”。
4. 再用模型做语义抽取，输出候选工具。
5. 将候选工具和 AgentTool 注册表匹配。
6. 未匹配的工具进入 `unresolved_tool_hints`，由管理员确认是否新增工具。
7. 用户确认后生成 AgentTool 绑定。

### 5.4 抽取不等于授权

抽取出的工具只是需求，不代表 Agent 可以调用。真正运行前仍必须通过：

- AgentBinding 是否启用。
- AgentRule 是否允许。
- 用户权限是否允许。
- 工具自身是否启用。
- 高风险工具是否需要二次确认。

## 6. 知识识别与核心提示词抽取

每个子 Agent 都可以维护自己的公众号、Markdown、文章链接和 Git 知识库。系统需要把这些非结构化内容变成可运行的 Agent 能力。

### 6.1 来源动态识别

新增知识时，系统先识别来源类型：

| 输入 | 识别结果 | 处理方式 |
| --- | --- | --- |
| `https://mp.weixin.qq.com/...` | `wechat_mp` | 保存原文链接、标题、作者、公众号名称，解析正文 |
| `.md` 文件 | `markdown` | 解析 front matter、标题、章节、代码块 |
| Git 仓库 | `git_repo` | 按目录规则扫描 agent package、Markdown、prompt 文件 |
| 普通 URL | `url_article` | 抓取标题、正文、发布时间和 canonical URL |
| PDF/附件 | `pdf` | 转文本后按章节切分 |
| 手写内容 | `manual_note` | 直接入库并标记人工来源 |

识别后写入 `AgentKnowledgeSource` 和 `AgentKnowledgeItem`。

### 6.2 Markdown 规范识别

Markdown 文章建议支持以下 front matter：

```yaml
---
title: 基金板块轮动分析方法
agent: fund_rotation_agent
source_type: methodology
tags: [fund, sector, rotation]
tools:
  - fund_holdings
  - sector_market_snapshot
rules:
  - cite_data_date
  - no_fabrication
license: private
---
```

如果没有 front matter，系统自动从标题、章节和关键词识别。

### 6.3 核心提示词抽取

核心提示词抽取不是简单摘要，而是把文章里的“方法论”和“行为约束”转成可执行提示词片段。

抽取目标：

- 角色定义：这个 Agent 应该以什么身份工作。
- 分析框架：先看什么、再看什么、如何下结论。
- 数据要求：必须读取哪些字段、日期、来源。
- 工具要求：需要调用哪些工具。
- 输出模板：结果应该如何组织。
- 风险约束：哪些话不能说，哪些结论要降级。
- 样例 few-shot：可复用的高质量输入输出例子。

抽取结果先写入 `AgentPromptExtraction`，状态为 `candidate`。用户确认后才能生成：

- AgentPromptVersion 的 prompt 片段。
- AgentRule 的规则。
- AgentSkill 的默认提示词。
- AgentTool 的候选工具需求。

### 6.4 抽取流水线

```text
Add Knowledge Source
  -> Detect Source Type
  -> Parse Content
  -> Split Sections
  -> Extract Entities
  -> Extract Core Prompt
  -> Extract Rules
  -> Extract Tool Hints
  -> Generate Candidates
  -> Human Review
  -> Publish to Agent
  -> Optional Export Package
```

### 6.5 去重和版本

- 同一 URL、同一文件路径、同一 content_hash 不重复入库。
- Git 来源记录 commit hash，支持对比两次提交的变化。
- 抽取结果按来源 item 版本追踪，原文变化后重新抽取。
- 已发布 prompt/rule 不静默覆盖，只能生成新版本。

### 6.6 版权和社区边界

- 公众号文章默认只能保存摘要、结构化抽取和来源链接，原文全文是否分发取决于授权。
- 私有 Markdown 可以完整进入私有 Agent，但导出社区包时需按 license 判断是否包含。
- 社区包默认只包含 prompt、rules、skills、tool requirements、knowledge index，不包含未授权全文。
- 每条抽取结果保留 source_ref，运行时可以展示“来自哪个公众号/哪篇 Markdown/哪个 commit”。

### 7.3 Git 提交流程

1. 用户在页面创建或更新 Agent。
2. 系统执行知识抽取和候选确认。
3. 用户点击“导出 Agent Package”。
4. 系统生成标准目录和 `agent.json`。
5. 运行 package 校验：schema、工具、规则、样例、敏感信息。
6. 通过后提交到 Git 分支或生成 PR。
7. 社区审核：版权、质量、安全、工具依赖。
8. 审核通过后发布为社区 Agent。

### 7.4 Package 校验

提交前必须校验：

- `agent.json` 必填字段完整。
- tools 都能在 AgentTool 注册表匹配，未匹配则标为 unresolved。
- rules 都存在或随包定义。
- prompts 不包含 API Key、Cookie、Token、私有账户信息。
- knowledge 不包含未授权全文。
- examples 能跑通最小测试。
- 输出模板包含来源、日期、风险提示。

### 7.5 社区安装

安装社区 Agent 时：

1. 拉取 package。
2. 校验 schema 和版本。
3. 导入 AgentDefinition、Prompt、Skill、Rule、Tool Hint。
4. 未安装工具进入待绑定状态。
5. 知识摘要进入 AgentContent，未授权全文不导入。
6. 用户选择模型配置后启用 Agent。

## 8. Agent Runtime

### 8.1 运行阶段

```text
Create Run
  -> Load Agent
  -> Check Permission
  -> Build Context
  -> Render Prompt
  -> Resolve Skills
  -> Resolve Rules
  -> Plan Tools
  -> Optional User Confirm
  -> Execute Steps
  -> Validate Result
  -> Write Audit
  -> Return Result
```

### 8.2 Context Builder

Context Builder 负责把业务数据组装给 Agent。它必须显式记录数据来源和日期。

常见上下文：

- 基金：基础信息、基金公司、基金经理、规模、净值、持仓、板块、排行、社区信息。
- 股票：基础信息、行情、K线、指标、公告、F10、资金流、行业板块。
- 用户：自选列表、持仓、账户、策略配置。
- 市场：板块涨跌、指数、资金流、排行榜。
- 内容：Agent 文章、方法论、历史样例。

### 8.3 Rule Engine

Rule Engine 分为运行前、运行中、运行后三类校验：

- 运行前：权限、预算、工具授权、输入参数。
- 运行中：工具超时、重试、是否需要用户确认。
- 运行后：输出格式、引用来源、风险提示、敏感信息。

### 8.4 Tool Planner

第一阶段不需要做复杂自主规划，建议采用受控规划：

- 用户选择 Skill 时，先用 Skill 的 required_tools。
- 提示词/文章抽取出的工具作为 optional_tools。
- Agent Rules 决定哪些工具可用。
- 模型可以建议工具，但后端只允许调用白名单工具。

### 8.5 Result Validator

输出必须经过校验：

- Markdown 输出：检查标题、来源、日期、风险提示。
- JSON 输出：按 output_schema 校验。
- 推荐类输出：必须写入推荐记录或明确标记为未入库草稿。
- 投资结论：必须包含不确定性和反向风险。

## 11. 权限与安全

### 11.1 权限边界

- 私有 Agent 仅创建者可见。
- 公共 Agent 可被其他用户调用，但不能读取创建者私有上下文。
- 系统 Agent 由管理员维护。
- 私有持仓、自选、交易日志进入上下文前必须校验用户。
- 高风险工具必须单独授权。
- 私有知识源默认不能进入社区包。
- 公众号全文默认不能随 Package 分发，除非 license 明确允许。
- Git 导入只允许访问用户授权的仓库和分支。

### 11.2 审计要求

必须记录：

- Agent ID、Prompt 版本、Rule 版本、Skill 版本。
- 用户输入摘要。
- 上下文数据来源和日期。
- 工具调用参数摘要、结果摘要、耗时、错误。
- 模型名称、Token、成本。
- 最终输出摘要。
- 用户确认、取消、重试、分享动作。
- 知识源同步记录、抽取记录、人工确认记录。
- Package 导出、导入、Git 提交、社区审核记录。

### 11.3 风控要求

- Agent 不允许直接发起真实交易。
- 推荐股票/基金必须标记为分析结果，不等于投资建议。
- 推荐入库必须保存输入数据版本和生成依据。
- 无来源数据不能编造。
- 对外分享前必须脱敏私有上下文。
- 社区 Agent 安装前必须展示工具依赖、数据权限和风险规则。
- 从外部导入的 Agent 默认禁用高风险工具。

### 12.1 复用现有能力

## 15. 关键决策

- Agent 是独立一级能力，不只是提示词模板。
- 每个子 Agent 都有自己的知识源，公众号、Markdown、Git、文章链接都是一等对象。
- 核心提示词、规则、工具需求可以从知识中抽取，但必须人工确认后才能发布。
- Agent Package 是社区流通标准，Git 提交和社区审核都基于 Package。
- 社区包不默认包含未授权全文和用户私有数据。
- Skill 是可执行能力，Prompt 只是 Skill 的一部分。
- Rules 是强约束，不能只写在提示词里。
- 工具抽取只产生候选，不自动授权。
- AgentRun 必须完整审计，尤其是投资分析和推荐类输出。
- 第一阶段采用受控工具规划，暂不允许模型自由调用任意工具。
