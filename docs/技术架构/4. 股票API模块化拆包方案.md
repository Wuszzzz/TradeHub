# 股票 API 模块化拆包方案

## 1. 背景

当前 `tradehub-stock/api` 已经具备健康检查、标的搜索、实时行情、历史行情、ETF 风控、采集任务、告警、模拟交易等能力，但代码仍偏工具型 API 结构。后续要承载技术指标、61 种 K 线形态、形态选股、策略回测、AI 研究中心、交易日志等功能，必须先拆清楚内部包结构。

目标不是把股票侧拆成完整微服务体系，而是在 `tradehub-stock` 一个大服务里拆出稳定模块边界。行情网关可以继续作为独立进程存在，量化、回测、AI 研究第一阶段先作为股票服务内部模块。

## 3. 分层职责

| 层 | 职责 | 不做什么 |
| --- | --- | --- |
| `interfaces/http` | 路由、参数解析、响应封装、鉴权、request_id | 业务计算、SQL、行情源调用 |
| `application/*` | 编排用例、事务边界、任务创建、调用领域服务 | HTTP 细节、第三方源协议 |
| `domain/*` | 领域模型、指标计算、形态识别、策略规则 | 数据库、HTTP、Redis |
| `infrastructure/*` | PostgreSQL、TDengine、Redis、market-api、AKShare 适配 | 业务规则 |
| `pkg/contract` | DTO、枚举、统一响应、错误码 | 具体业务实现 |

## 4. 模块拆分

### 4.1 instrument

职责：

- 标的搜索。
- 标的资料。
- 市场、交易所、板块、行业归一化。
- 股票、ETF、指数、基金等资产类型识别。

应用服务：

```text
SearchInstruments(keyword, market, assetType)
GetInstrumentProfile(symbol)
SyncInstrumentMetadata(symbols)
```

### 4.2 market

职责：

- 实时快照。
- K 线。
- 分时。
- 五档盘口。
- 资金流。
- 价位成交分布。
- 行情源降级。

应用服务：

```text
GetSnapshot(symbol)
GetKline(symbol, period, adjust, start, end, limit)
GetOrderBook(symbol)
GetMoneyFlow(symbol)
GetPriceDistribution(symbol)
```

### 4.3 watchlist

职责：

- 股票自选分组。
- 自选排序。
- 标签。
- 自选触发采集任务。

应用服务：

```text
CreateGroup(userID, name)
AddItem(userID, groupID, symbol)
RemoveItem(userID, groupID, symbol)
ListItems(userID, groupID)
```

### 4.4 quant

职责：

- 技术指标定义。
- 指标计算。
- 61 种 K 线形态定义。
- 形态扫描。
- 指标选股。
- 形态选股。

应用服务：

```text
ListIndicators()
CreateIndicatorDefinition(input)
ComputeIndicator(symbol, period, indicator, params)
ListPatterns()
CreatePatternDefinition(input)
ScanPattern(symbols, period, patterns, params)
RunScreener(filter)
```

### 4.5 backtest

职责：

- 策略参数。
- 回测任务。
- 回测结果。
- 交易明细。
- 绩效指标。

应用服务：

```text
CreateBacktest(strategyID, params, universe, dateRange)
GetBacktestResult(taskID)
ListBacktestTrades(taskID)
CompareBacktests(taskIDs)
```

### 4.6 alert

职责：

- 告警规则。
- 告警扫描。
- 事件确认。
- 冷却时间。

应用服务：

```text
CreateRule(userID, rule)
ListRules(userID)
ScanRules()
AckEvent(userID, eventID)
```

### 4.7 paper

职责：

- 模拟账户。
- 模拟持仓。
- 模拟订单。
- 成交与资金流水。
- 收益统计。

应用服务：

```text
GetAccount(userID)
SubmitOrder(userID, order)
ListPositions(userID)
ListOrders(userID)
ResetAccount(userID)
```

### 4.8 task

职责：

- 统一任务创建。
- 任务状态机。
- 重试。
- 进度。
- 日志。
- 取消。

应用服务：

```text
CreateTask(type, params)
StartTask(taskID)
UpdateProgress(taskID, progress)
SucceedTask(taskID, resultRef)
FailTask(taskID, error)
CancelTask(taskID)
```

## 5. 迁移步骤

### 第 1 步：抽统一响应

新增：

```text
internal/interfaces/http/response.go
internal/interfaces/http/errors.go
internal/interfaces/http/middleware.go
```

目标：

- 所有新接口使用统一响应。
- 老接口先不强改，避免前端同时大改。

### 第 2 步：抽 repository 接口

在 application 层依赖接口，不直接依赖 `psql` 命令或 TDengine HTTP 细节。

```go
type InstrumentRepository interface {
    Search(ctx context.Context, query InstrumentQuery) ([]Instrument, error)
    Get(ctx context.Context, symbol string) (*Instrument, error)
}
```

### 第 3 步：迁移 market 模块

先迁移实时快照和 K 线，因为后续指标、形态、回测都依赖它。

旧路径保持：

```text
/api/v1/market/realtime
/api/v1/market/history
/api/v1/market/daily
```

新增路径：

```text
/api/stock/v1/market/snapshot
/api/stock/v1/market/kline
```

### 第 4 步：迁移 alert 和 paper

告警和模拟交易相对独立，适合在 market 之后迁移。

### 第 5 步：新增 quant 和 task

指标、形态、选股、回测都要先接入 task，而不是直接写同步接口。

## 6. 兼容策略

| 现状 | 策略 |
| --- | --- |
| `api/main.go` 中已有路由 | 先迁移到 router，旧路径保持 |
| `PostgresRepository` 使用 `psql` 命令 | 第一阶段可保留实现，但通过 repository interface 隔离 |
| Python AKShare 子进程 | 保留为 fallback，CN 行情优先走 market-api |
| 前端已有接口调用 | 不破坏，新增路径后逐页迁移 |

## 7. 测试策略

每迁移一个模块补三类测试：

| 测试 | 内容 |
| --- | --- |
| domain test | 指标、形态、策略规则纯函数测试 |
| application test | 用 mock repository 测业务编排 |
| http test | 参数、响应、错误码、兼容路径 |

## 8. 第一批拆包优先级

优先级：

1. `response/errors/middleware`
2. `market`
3. `instrument`
4. `watchlist`
5. `task`
6. `alert`
7. `paper`
8. `quant`
9. `backtest`
10. `ai`

这个顺序能先稳定数据入口，再承接技术指标、K 线形态和选股回测。
